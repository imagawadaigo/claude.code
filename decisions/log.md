## 2026-04-07：リポジトリの分け方ルール策定

**決定事項**
- `claude.code` = 思考・企画・判断・ToDo（頭の中）。デプロイしないものはすべてここ
- プロダクトのコード = プロダクトごとに別リポジトリ（例：`flower-portrait`）
- 別リポジトリを作るタイミング = 「コードを書き始める」瞬間。企画段階は `claude.code` 内で育てる
- 判断基準：「npm start や vercel deploy で動かすものか？」→ Yes なら別リポジトリ

## 2026-04-07：リポジトリ構造の整理

**背景**
- チャットごとにブランチが作られ、ナレッジが断片化していた
- GitHubに潜らないと何がどこにあるか分からない状態だった

**決定事項**
- ドキュメント・思考整理 → mainに直接コミット
- コード開発 → feature/xxx ブランチ → main にマージ
- CLAUDE.md を「全体の地図」として機能させ、全ファイルを@インポートする
- `decisions/log.md` を追記型で運用し、判断の経緯を残す

**ディレクトリ構造**
```
claude.code/
├── CLAUDE.md               ← 毎セッションの入口・全体地図
├── businesses/
│   ├── philosophy.md       ← 思想・判断軸
│   └── strategy.md         ← 事業戦略（将来追加）
├── docs/
│   └── mothers-day-service.md  ← プロダクト仕様
└── decisions/
    └── log.md              ← この判断ログ
```

## 2026-04-07：プロダクト方向性の決定

**決定事項**
- 母の日メッセージ花診断サービスをMVPとして開発
- AIの役割は花の診断・分類のみ（メッセージ代筆はしない）
- リンク生成ごとに課金するモデル
- 今年はテスト運用、来年本番

**理由**
- 「伝えたい気持ちの橋渡し」というビジョンとの一貫性
- 低維持費で早期収益化が狙える
- BtoC → BtoB への波及設計が可能
